# Troubleshooting Runbook (5-Why Framework)

**Trigger:** เมื่อ Agent (AI) รัน Test ไม่ผ่าน (แบบผิดแผน), เกิด Architecture Violation, หรือเจอบั๊กที่แก้ไม่ได้ภายใน 1 ครั้ง

## ขั้นตอนการทำงานสำหรับ Agent (SOP)

เมื่อเข้าสู่ภาวะ Troubleshooting ห้าม Agent หาวิธีแก้ด้วยการสุ่มโค้ดเด็ดขาด ให้ทำตามลำดับต่อไปนี้:

1. **หยุดและรวบรวมหลักฐาน (Stop & Collect Evidence)**
   - อ่าน Error Log อย่างละเอียด อย่าเดาเอง
   - หากข้อมูลไม่พอ ให้ขออนุญาตผู้ใช้ (User) รันคำสั่งเพิ่มเพื่อดึง Log ออกมา

2. **วิเคราะห์หารากของปัญหา (5-Why Analysis)**
   แสดงผลการวิเคราะห์ให้ User ดูโดยใช้รูปแบบ 5-Why:
   - *Why 1:* ทำไมคำสั่งถึง Fail? (เช่น error message คืออะไร)
   - *Why 2:* ทำไมมันถึงเกิด error นั้น? (โยงกลับไปที่บรรทัดโค้ด)
   - *Why 3:* ทำไมโค้ดบรรทัดนั้นถึงทำงานผิด? (เช่น ค่าที่ส่งมาเป็น null)
   - *Why 4:* ทำไมค่าถึงเป็น null? (เช่น ลืม Mock)
   - *Why 5:* สาเหตุที่แท้จริง (Root Cause) คืออะไร?

3. **เสนอทางแก้ (Propose Solutions)**
   - นำเสนอวิธีแก้ให้ User 
   - รอ User อนุมัติ (ห้ามเขียนโค้ดแก้ก่อนได้รับ Approve)

4. **ลงมือแก้ไขและยืนยันผล (Execute & Verify)**
   - เมื่อแก้ไขแล้ว ต้องรัน Sensor (เช่น `vitest` หรือ `m-hex.py`) เพื่อยืนยันว่าปัญหาถูกแก้จริง และอัปเดตลง `lesson-learned` หากเป็นบั๊กที่น่าสนใจ
